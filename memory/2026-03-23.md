## 重要教训：pharma-report-analyzer 技能迭代准则

### 事件回顾
今天对 pharma-report-analyzer skill 进行了 v2.1.0 → v2.2.0 的实验性升级尝试，内容包括：
1. **流式提取引擎**：用 `iter_block_items` 按文档原文顺序还原文字+表格
2. **强制归档协议**：在 `analyze` 阶段自动复制原始 .docx 到报告原文目录
3. **自动化逻辑链推导**：试图让脚本自动生成 DP-C 逻辑关系

### 结果
- 用户两次反馈"**还是不行**" / "**完全不对了**"
- 核心问题：过度关注底层代码变动，导致分析内容深度严重退化（汇总变薄、问题清单缺失）
- **最终方案**：完整回滚至 v2.0.0 稳定版

### 关键教训
| 教训 | 说明 |
|------|------|
| **分析深度不可妥协** | 任何技能优化都必须保证输出内容比之前更完整，绝不能用占位符或简化版 |
| **实验分支需独立验证** | 新版本应在独立目录验证后再合入主框架 |
| **用户反馈优先响应** | 连续两次"不行"表明方向性错误，应立即回滚而非继续迭代 |

### 未来优化前提
如需升级提取逻辑（如文档流顺序），必须满足：
- 在保持 v2.0.0 分析深度（简要汇总/逻辑表/问题清单）的前提下局部改进
- 必须在独立测试中验证通过后再合入
- 不得用占位符内容应付分析输出
