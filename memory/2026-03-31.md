## 本次主要工作：memory_search 向量索引重建

---

### 问题症状
memory_search 搜不到任何内容，返回空结果。

---

### 诊断过程（历经 4 小时）

**初步诊断**：
- SQLite 向量索引数据库 `main.sqlite` 是空的（0 chunks），从未被填充
- Ollama 运行正常，embedding 生成正常（nomic-embed-text）
- Gateway 的 memory sync 从未成功完成过

**发现的多个根本原因**：

| # | 问题 | 影响 | 状态 |
|---|------|------|------|
| 1 | **fallback 配置错误**：Ollama 失败后 fallback 到 gemini，但无 API key 返回 405 | provider=null，索引管道完全中断 | ✅ 已修复（fallback: "none"） |
| 2 | **batch timeout 太短**：remote provider timeout 仅 120s，Ollama 冷启动需 1-5 分钟 | 批处理始终超时 | ✅ 已修复（600s） |
| 3 | **snowflake-arctic-embed:33m 在 Ollama 中会立即卡死**：runner 进程 184% CPU 无响应，无法完成任何 embedding 调用 | 切换该模型后完全无响应 | ✅ 已回退（改回 nomic-embed-text） |
| 4 | **Ollama runner 会意外卡死**：长时间运行后 nomic-embed-text runner 可能进入不可恢复的挂起状态（CPU 100%+ 无响应）| 需手动重启 Ollama | ⚠️ 待观察 |

**修复的配置变更**（均通过 config.patch 写入）：
```yaml
agents:
  defaults:
    memorySearch:
      fallback: "none"          # 原来无值（auto），导致 gemini 405
      remote:
        batch:
          enabled: true
          concurrency: 1       # Ollama 单连接最稳
          timeoutMinutes: 10   # 120s → 600s
      chunking:
        tokens: 200            # snowflake 模型需要更小 chunk（已无用，但保留）
```

---

### 最终方案（2026-03-31 13:00+）

由于 `openclaw-memory index --force` CLI 在 Ollama 环境下持续崩溃，采用了**手动构建向量索引**的临时方案：

1. 直接写 SQL：清空并重建 `chunks` 表（277条 chunk，35个 memory 文件）
2. 直接写 SQL：用 JSON 字符串填充 `embedding` 列（绕过 Ollama API）
3. **不创建 `chunks_vec` 表**（sqlite-vec 虚拟表只能 gateway 运行时创建）
4. **依赖 OpenClaw 内置的 fallback**：`chunks_vec` 不存在时自动回退到 `chunks.embedding` 列的纯 Python 余弦相似度计算（实测可用）

**当前索引状态**：
- `chunks`: 277 条 ✅
- `embedding_cache`: 366 条（provider_key 用哈希值，非 "ollama"）✅
- `chunks_vec`: 不存在（回退到 chunks.embedding 列搜索）✅
- 搜索功能：**可用**（纯 Python fallback 模式）

---

### 遗留问题

1. **Gateway 重启后 `chunks_vec` 仍不会被自动创建**——因为 embedding 阶段会超时。需要修复 Ollama 稳定性或改用更快的 embedding 模型。
2. **Ollama runner 卡死问题**：长时间运行后需要监控，疑似内存泄漏或 runner 管理问题。
3. **推荐后续方案**：
   - 短期：保持 nomic-embed-text + 手动索引
   - 中期：调查 Ollama nomic-embed-text 为何冷启动需 17s（可能是模型加载问题）
   - 长期：考虑用 OpenAI API key 或本地更小的 embedding 模型
